```bash
# Backend: PostGIS + migrations + app
cd backend && docker compose up -d --build
curl localhost:8080/api/v1/health

# Frontend (points at the backend via NEXT_PUBLIC_API_URL, default :8080)
cd frontend && pnpm install && pnpm dev   # http://localhost:3000
```

   **Why building it would have made things worse.** The cache would have moved event-detail traffic *off* the uncapped JS API and *onto* the Static API, which **is** capped — §6.3.1: «Количество обращений к Static API не должно превышать 1 000 (Одну тысячу) Запросов в сутки». It would have introduced a quota ceiling where none existed, plus a new key, a new cabinet product, and a cache to maintain.

The plan's "cache permanently, ~11 requests forever" directly contradicts this. Any future caching of Yandex responses must carry a **≤30-day TTL**.

   **The real quota exposure is the Geocoder**, §6.6.2: «не должно превышать суммарно **1000 (Одну тысячу) Запросов в сутки**» — that's what the auth-gated `/api/v1/geocode` proxy hits. If any map quota work is ever needed, cache *geocode results per address* (≤30-day TTL). Note §6.6.1 also requires Geocoder results be used only for display via the JS API — storing coords in our DB is defensible only because they render on a Yandex map; don't surface them elsewhere.

VK Maps issues keys only on signing a contract. Sber's maps product *is* 2GIS. ## Known gotchas (don't re-discover these)

- **Yandex Maps — the whole trap list** (cost hours on 2026-07-16):
  - **`NEXT_PUBLIC_*` needs an `ARG`/`ENV` in `frontend/Dockerfile` before `RUN pnpm build`.** Docker **silently ignores** a `--build-arg` for an undeclared `ARG`, so the key inlines as `""` and every map renders the "Карта недоступна" placeholder in prod — a green build that ships a mapless site. Applies to `NEXT_PUBLIC_YANDEX_MAPS_KEY` *and* `NEXT_PUBLIC_API_URL`.
  - **JS API v3 does NOT work with our key** — `api-maps.yandex.ru/v3` returns `403 {"message":"Invalid api key"}` even with the Referer restriction set and propagation time; **v2.1 serves the same key fine**. Root cause never confirmed. We ship v2.1. Don't "upgrade to v3" without new information from Yandex.
  - **A JS API key must have an HTTP Referer restriction configured or it stays "Не настроен"** and is rejected. Format = **bare domains, no protocol/port/url, one per line** (`presence.tarski.ru`, `localhost`). The key-config screen exists **only in the new cabinet UI**: `developer.tech.yandex.ru/keys?modern=true` → Ключи → ⋯ → Настроить.
  - Because keys are Referer-restricted, a bare `curl` is **correctly** rejected — always test with `-H "Referer: https://presence.tarski.ru/"`.
  - **JS API map loads are FREE and UNCAPPED** — «Запросы к карте (hits) являются бесплатными» ([tariffs](https://yandex.ru/dev/tariffs/doc/ru/jsapi/prices)). An earlier note here claimed "free JS API = 100 req/day"; **that was wrong** — §6.1.2's 100/day governs **Показ Панорам**, which we don't use. It caused a whole cached-static-map project to be planned against a limit that doesn't exist (see "What's next" #5, cancelled). The **real** caps: **Geocoder 1 000/day** (§6.6.2 — this is our actual exposure), **Static API 1 000/day** (§6.3.1), Tiles API 30 rps with no daily cap (§6.5.2), MapKit 1 000 *unique users*/day (§6.2.3). Verify limits against [yandex.ru/legal/maps_api](https://yandex.ru/legal/maps_api/) §6 before planning around them — read which *service* each clause names.
  - **Caching Yandex responses is capped at 30 days** — §5.1.5 permits «временного хранения (кэширования)… на срок не более 30 дней» and nothing longer. Never design a permanent cache of Yandex data.
  - **§4.1.3.3 requires an on-screen "Открыть в Картах" button**, and §4.1.3.1/§5.1.2 forbid moving/hiding/cropping/overlaying the Yandex copyright and logo. This is a licence condition, not a design preference.
  - **Coordinate order is a live footgun**: our props/`MapPin` are `[lat, lon]`; **v2.1 is `[lat, lon]` too** (no conversion — this is why `toLngLat` was deleted), but **v3 is `[lon, lat]`**, the Geocoder's `Point.pos` is **`"lon lat"`**, and Yandex Maps deep-links use **`?pt=lon,lat`**. Get it wrong and pins land in the wrong country.
  - Venues with no `lat`/`lon` render **no map at all** by design (`EventDetailView.tsx:130`) — that's not a bug.
- **The box's `/opt/lia/backend` is NOT a git checkout** (scp'd files, and stale — it lacks `internal/geocode`). Images are built on Mac (`--platform linux/amd64`, tags `amd64-rN`) and shipped `docker save | gzip | ssh | docker load`. Backend compose needs **all four** files (`docker-compose.yml` + `.prod.yml` + `.gateguard.yml` + `.monitoring.yml`) + `--env-file .env.prod`; use `up -d --no-build` or compose may rebuild the box's stale source. Frontend is a plain `docker run` (`127.0.0.1:3002→3001`), not compose.
- **Template codegen**: after `make rename`, run `make generate-all` (go-swagger server + protobuf) before `go build` — that code is gitignored and regenerated in CI.
- **`rename.sh` regex**: the template's rejected dots/uppercase; relaxed to accept full Go module paths. Dockerfile binary-copy path and the `COPY .git` line were also fixed for the monorepo-subdir layout.
- **go-pg + gofrs UUID**: cannot scan SQL `NULL` into a uuid field. "Unset" organizer/venue is the **zero UUID** (`NOT NULL DEFAULT`), and `events.Create` avoids `RETURNING *`. (Nullable non-uuid columns like `venues.lat`/`lon` are fine as `*float64`.)
- **go-pg raw `Query` skips hooks**: `events.repository.Nearby` scans events via raw SQL (for the PostGIS `distance_m`), so `AfterSelect` (which maps `StatusSQL`→`Status`) must be called manually per row. Prefer the model-based load when you don't need raw SQL — `loadVenues` uses it so nested-venue columns can't silently drift.
- **swagger nullable fields**: declare optional `number` fields (e.g. coords, `distance_m`) with `x-nullable: true` so go-swagger generates `*float64` (+ `omitempty`) — otherwise unset values serialize as `0` and a partial `PATCH` zeroes stored data. Create vs update use distinct bodies (`VenueInput` requires `name`; `VenueUpdateInput` doesn't).
- **golangci-lint**: CI installs **v1** (the `.golangci.yml` is v1 format) — do **not** migrate it to v2. Locally, install v1 to lint as CI does.
- **Prod `app` recreate needs ALL THREE compose files**: `-f docker-compose.yml -f docker-compose.prod.yml -f docker-compose.gateguard.yml`. The gateguard file is the ONLY place the `app` service gets `HTTP_GATEKEEPER_ADDRESS=gateguard:9090` + `HTTP_MOCK_AUTH` (viper `AutomaticEnv`, `cmd/root/root.go`; no in-code env binding for the gatekeeper address — default `localhost:9091`). Recreate without it → register/login + token validation 503 (signer dials `127.0.0.1:9091`) and `HTTP_MOCK_AUTH` reverts to base dev default `true`. Caused a live incident 2026-06-26.
- **go-pg `use_zero` defeats DB column defaults**: a field tagged `pg:"col,use_zero"` writes the Go zero value (e.g. `""`) on INSERT instead of omitting the column, so a `DEFAULT 'x'` never applies.
