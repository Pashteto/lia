# Email verification (SendPulse) + event invitations — Deploy

_2026-07-16. As executed. Target: `presence.tarski.ru` / `api.presence.tarski.ru`
on vds-ru215 (`ssh vdska2`, 193.32.188.7). Migration **019 → 020**._

**Outcome:** deployed and healthy. `tarski.ru` verified in SendPulse (DKIM+SPF+DMARC
green). Not yet proven: that a verification email actually lands in an inbox — that
needs a real signup (see "Remaining" below).

## What shipped

- **GateGuard**: 6-digit verification codes emailed via SendPulse SMTP (`NOTIFICATOR_*`).
- **Backend**: event invitations + verified-gating; invite emails via `SMTP_*`;
  `PUBLIC_BASE_URL` builds the `/invite/<token>` links. Migration `000020_event_invitations`.
- **Frontend**: `/auth/verify`, `/invite/[token]`, `/me/invitations`.

## Traps this deploy hit (read first — each cost real time)

1. **The box's `/opt/gateguard` was STALE.** It held upstream + an *old stub* that
   logged the verification link instead of emailing it. GateGuard is NOT built from
   `1-lia/gateguard` by default — the old runbook rsyncs it from
   `~/gateway_fm/appstore/gateguard` (a different checkout, which has no
   `email_verification.go` at all). **Sync `1-lia/gateguard/` → box `/opt/gateguard/`
   and verify by md5 before building.**
2. **Building gateguard ON the box FAILS.** `curl` to github.com dies with
   `TLS connect error ... unexpected eof` fetching protoc-gen-grpc-web. The
   Dockerfile's `-4 --retry 5 --retry-connrefused` does NOT help (it doesn't retry
   TLS errors). **Build gateguard on the Mac instead** — same as backend/frontend:
   ```
   cd 1-lia/gateguard && docker build --platform linux/amd64 -t gateguard:new .
   docker save gateguard:new | gzip | ssh vdska2 'gunzip | docker load'
   ```
   Confirm `docker image inspect gateguard:new --format '{{.Architecture}}'` = `amd64`.
3. **`.env.prod` alone is NOT enough.** Compose only passes variables it *declares*.
   Both compose files needed edits: `NOTIFICATOR_*` into `docker-compose.gateguard.yml`'s
   `gateguard.environment`, and `SMTP_*` + `PUBLIC_BASE_URL` into
   `docker-compose.prod.yml`'s `app.environment`. Verify with
   `docker compose ... config | grep -E 'NOTIFICATOR_|SMTP_'`.
4. **`docker exec <backend> env` returns NOTHING** — the image is scratch-based, no
   shell/`env` binary. That is NOT proof the env is missing. Use
   `docker inspect <c> --format '{{range .Config.Env}}{{println .}}{{end}}'`.
5. **Container/image names are not what you'd guess.** The live images are
   `backend-app:latest` (compose `build: .`, no `image:` key) and
   `lia-frontend-presence:latest` — NOT `lia-backend:latest`/`lia-frontend:latest`.
   Tagging rollbacks against the wrong names silently no-ops.
6. **The frontend is NOT run by compose** — it's a bare
   `docker run -d --restart unless-stopped --name lia-frontend-presence -p 127.0.0.1:3002:3001`.
   Ignore `/opt/lia/frontend-build.sh`: it is **stale** and builds with
   `NEXT_PUBLIC_API_URL=https://api.lia.pashteto.com`, which degrades the site to mock data.
7. **`dig`/`host` do not resolve from the Claude sandbox at all.** Verify DNS via DoH:
   `https://dns.google/resolve?name=<n>&type=TXT`.

## Procedure (as executed)

1. **Backup**: `pg_dump | gzip > /opt/lia/backup-pre-email-invites-<ts>.sql.gz`.
2. **Sync gateguard source** `1-lia/gateguard/` → `vdska2:/opt/gateguard/`; md5-verify
   `internal/service/email_verification.go` matches local.
3. **Build gateguard on Mac** (trap 2) → save/load to box.
4. **Env**: back up `.env.prod`, append `NOTIFICATOR_{ADDRESS,USERNAME,PASSWORD,FROM}`,
   `SMTP_{ADDRESS,USERNAME,PASSWORD,FROM}`, `PUBLIC_BASE_URL=https://presence.tarski.ru`;
   `chmod 600`. Values → the git-ignored `2026-07-15-sendpulse-nicru-credentials.local.md`.
   Both use `smtp-pulse.com:587` (2525 is the fallback if the network blocks 587).
5. **Wire compose** (trap 3), then `docker compose ... config` to confirm resolution.
6. **Migration 020**: `scp` both `.sql` to `/opt/lia/backend/db/migrations/`, then run
   `/opt/lia/migrate-020.sh` (on the box; prints version before/after + table check).
   Expect `19 → 20`, `dirty=f`, `to_regclass = event_invitations`.
   > Write this as a **script file**, don't paste a long one-liner — a wrapped
   > `docker run ... \n -path=/db/` makes `migrate` print usage and bash then tries to
   > execute `-path=/db/`. And use **single quotes** in SQL: `" dirty="` is read by
   > Postgres as a *column identifier*, not a string.
7. **Backend**: `make generate-api` FIRST (swagger `internal/http/{models,server}` are
   git-ignored and NOT regenerated by `make build`; without it `email_verified` is
   missing from the built code — confirm with `grep -rl EmailVerified internal/http/models/`).
   Then build amd64 → ship → `docker tag lia-backend:email-r1 backend-app:latest` →
   `docker compose --env-file .env.prod -f docker-compose.yml -f docker-compose.prod.yml
   -f docker-compose.gateguard.yml -f docker-compose.monitoring.yml up -d --no-build app`.
8. **GateGuard**: `docker tag gateguard:new gateguard:local` → same compose invocation
   with `--force-recreate gateguard`.
9. **Frontend**: build on Mac with **both** build-args
   (`NEXT_PUBLIC_API_URL=https://api.presence.tarski.ru` +
   `NEXT_PUBLIC_YANDEX_MAPS_KEY` from `frontend/.env.local`) → ship → tag
   `lia-frontend-presence:latest` → stop + **rename** the old container (keeps rollback)
   → `docker run` the new one on `127.0.0.1:3002:3001`.
10. **Verify** (all returned 200): `/`, `/auth/verify`, `/me/invitations`, `/map`,
    `api…/api/v1/events?limit=1`. Check gateguard logs for gRPC :9090 + no DB/redis errors.
    Confirm env via `docker inspect` (trap 4).
11. **Prune**: `docker builder prune -f` + `docker image prune -f` (reclaimed ~1.3 GB;
    75% → 68%). Trim `rollback-*` tags to the last ~3 — there are currently ~8 on
    `backend-app` and the disk is 20 GB.

## Rollback

  recreate with the 4 compose files `--no-build`.
- **GateGuard**: `docker tag gateguard:rollback-20260716-2129 gateguard:local` → recreate.
- **Frontend**: `docker rm -f lia-frontend-presence && docker rename
  lia-frontend-presence-old-20260716 lia-frontend-presence && docker start lia-frontend-presence`
  (the pre-deploy container was preserved, not deleted).
- **DB**: `migrate ... down 1` (020 → 019), or restore
  `/opt/lia/backup-pre-email-invites-20260716-2013.sql.gz`.
- **Env**: `.env.prod.bak-pre-email-*` and `docker-compose.*.yml.bak-pre-email-*` on the box.

## Remaining / caveats

- **NOT PROVEN: a verification email actually arriving.** GateGuard resolves
  `notificator.*` via `viper.AutomaticEnv()` + `Unmarshal`, and viper is known not to
  always feed Unmarshal from AutomaticEnv. Every key has a `SetDefault` so it *should*
  bind, but a miss **fails silently** back to Gateway.fm defaults
  (`smtp.gmail.com:587`, `infra@gateway.fm`). The env vars ARE in the container
  (verified) — what's unverified is that viper reads them. **A real signup to a real
  inbox is the only proof.** Check "show original" → DKIM=pass, SPF=pass, DMARC=pass.
- **Free tier: 50 emails/hour**, 12k/month, plan active until 2026-08-15. A burst of
  invitations larger than 50 will throttle.
- **Two deferred bugs, user-visible** (handoff §7): invite expiry not enforced on the
  accept path; re-inviting an already-pending email sends a dead link that 404s
  (`ON CONFLICT DO NOTHING` doesn't store the freshly-emailed token).
