## 1. Регистрация: второе поле — «Имя», а не повтор почты
`frontend/components/AuthButton.tsx` (`LoginModal`). Порядок полей в режиме регистрации: **Email → «Имя (необязательно)» → Пароль**. Поля повтора email нет — по дизайну (почта проверяется потом 6-значным кодом). Второе поле подписано корректно («Имя (необязательно)», placeholder «Ваше имя», без `type="email"`), но стоит сразу под Email → тестер на автомате прочитал его как «подтверждение почты».

**Вывод:** не баг, а несоответствие ожиданию. Дешёвое улучшение — визуально отделить «Имя» от блока учётных данных. Повтор email не нужен.

## 2. Известные площадки не находятся
Поиск площадки (`VenueGeoModal.tsx`, placeholder «Введите **адрес**») → `lib/geocode.ts` → `GET /api/v1/geocode` → **Yandex Геокодер** (`geocode-maps.yandex.ru/1.x/`). Геокодер резолвит **адреса (топонимы)**, а не названия организаций. «Дом Радио», «Ton Center», «NoDom» — это POI (организации), не адреса. Отдельного поиска по названиям (Yandex **Places/Search API**, `search-maps.yandex.ru/v1/?type=biz`) в проекте нет; локальной базы площадок для поиска тоже нет.

**Вывод:** архитектурное ограничение. Чтобы искать по названиям заведений — новый эндпоинт-прокси к Yandex Places API (отдельный ключ), как fallback/мёрдж с геокодером. Это фича, не почин.

## 3. Ссылку на «Bla Bla Meet» не открыть
Роут `frontend/app/events/[id]/page.tsx` (по UUID), гварда авторизации нет. Но бэкенд (`events.go:134`) отдаёт **404 всем, кроме владельца**, для любого статуса ≠ `published` (`draft`, `pending_review`, `rejected`, **`cancelled`**). Фронт превращает 404 в `notFound()`.

**Вывод:** если «Bla Bla Meet» было снято/черновик/на модерации — ссылка резолвит id, но бэкенд отдаёт 404 не-владельцу → «не найдено». Ожидаемое поведение видимости. *(Рядом единственный настоящий дефект роутинга: `app/invite/[token]/page.tsx:25` — fallback на пустой id → `/events/`.)* Связано с №5b.

## 4. Календарь: нет Google + грузится долго
**Google-интеграции нет by-design** — ни OAuth, ни «Добавить в Google», ни синка (`ics.go`: *«no OAuth, per spec»*). Единственный экспорт — скачивание `.ics`, и оно есть **только на CTA события** (`SignupCTA.tsx`), а на `/me/calendar` его нет.

**Slow-load:** каждая навигация по месяцам → свежий fetch (React Query без `staleTime`/prefetch), а бэкенд на каждый запрос делает **3 последовательных запроса** (`follows/handler.go:153` — 2 range-скана + полное переобогащение `GetEnriched`) без серверной пагинации. Диапазон месяца ограничен (42 дня), так что дело в fan-out + отсутствии кэша; VPN/удалённость усугубляют.

**Вывод:** «интеграция в Google» не поддерживается. Быстрый фикс: явно назвать «Скачать .ics», вынести на `/me/calendar`; синк с Google = фича. Скорость: `staleTime`/prefetch на фронте + схлопнуть 3 запроса на бэке.

## 5. Верификация почты
### 5a. Требуется даже у пришедшего по ссылке из письма
`email_verified` ставится в `true` **ровно в одном месте** — `gateguard/internal/service/email_verification.go:116`, только после ввода правильного 6-значного кода. Клик по инвайт-ссылке почту **не** верифицирует; более того, приём приглашения сам *требует* уже подтверждённую почту:
- `backend/internal/invitations/service.go:205` → `ErrNotVerified` если `!verified`;
- `gateguard/internal/service/react_to_invitation.go:75` → `ErrEmailNotVerified` при `Accepted`;
- фронт `app/invite/[token]/page.tsx:49` заменяет «Принять» на блок «Подтвердите почту».

**Вывод:** факт «письмо ушло на этот адрес, человек кликнул» логически *является* доказательством владения почтой, но система его игнорирует — это и есть «плохой flow». **Фикс:** сделать подписанную ссылку из письма (инвайт-токен) верифицирующей — при переходе ставить `email_verified=true` адресату.

### 5b. Баг с неподтв. почтой на «снятых» событиях
Ни один **read**-эндпоинт не гейтится верификацией. Проблема во фронте: `SignupCTA.tsx` **status-agnostic** — рисует активную кнопку «Записаться» даже для `cancelled`/`rejected` (ветвится только по `signupMode`, не по `event.status`) и на маунте дёргает `fetchEventWithAuth`. Взаимодействие бьёт в RSVP-эндпоинт, который verified-gated (`rsvp.go:42`) → `EMAIL_NOT_VERIFIED` → интерстишл «Подтвердите почту» на снятом событии, где записи быть не должно.

**Фикс:** `SignupCTA` должен ветвиться по `event.status` — для `cancelled`/`rejected` показывать «Событие отменено/снято» вместо CTA.

**Связанный над-широкий гейт:** `PATCH /events/{id}` (`events_update.go:33`) отклоняет **все** апдейты неподтверждённого пользователя до проверки владельца — неподтверждённый организатор не может даже отозвать своё событие. Стоит разрешить владельцу переводить своё событие в `cancelled` без верификации.

## 6. Баннер «Почта не подтверждена» висит даже у верифицированного пользователя / админа

> Заявлено пользователем 2026-07-23. Разобрано прямым чтением кода (первичная гипотеза «устаревший JWT-claim» **опровергнута**).

### Опровержение гипотезы «stale JWT»
`email_verified` **не** берётся из JWT. `gateguard/internal/service/check_auth.go:14-24` достаёт из токена **только UUID**, затем `UserByUUID` → `repository.GetUser` **читает пользователя из БД заново на каждый запрос**. Флаг сериализуется через proto field 13 (`gateguard/internal/models/user.go:109`), проходит `backend/.../gatekeeper.go:74` → `ensureUser` (`auth.go:143`) → `/auth/me`. **Кэша и вшитого claim'а в этом пути нет.** Значит, сразу после `VerifyEmail` следующий `/auth/me` вернёт `true`. **Перевыпуск токена (Вариант A) ничего не чинит** — токен этот флаг никогда не нёс.

### Реальная причина №1 (frontend — это и есть симптом)
`frontend/components/VerifyEmailBanner.tsx:19`:
```tsx
if (!ready || !isAuthed || emailVerified) return null;
```
`ready` становится `true` в момент **гидрации** (`getReadySnapshot` = `typeof window !== "undefined"`) — то есть **до** того, как `getMe()` вернётся. В этом окне `emailVerified` ещё в дефолте `useState(false)` → **баннер показывается на каждой жёсткой загрузке даже у верифицированного пользователя.** На медленном/флаки prod (VPN, удалённый бокс) окно длинное и заметное; а если `getMe()` **упадёт**, `catch` ставит `emailVerified=false` и баннер **зависает навсегда**. Комментарий в коде («`ready` предотвращает мелькание») ошибочен — `ready` не привязан к фетчу.

**Фикс:** гейтить по `roleResolved` (истина только после того, как `/auth/me` отработал), а не по `ready`:
```tsx
const { isAuthed, ready, roleResolved, emailVerified } = useAuth();
if (!ready || !isAuthed || !roleResolved || emailVerified) return null;
```

### Реальная причина №2 («даже если я админ»)
Админ `poulissimo` создан до появления верификации → его колонка `email_verified` реально `false`, а обхода для админа нет нигде (`verified_gate.go:15`, баннер). Для такого аккаунта баннер технически **корректен** — нужен бэкофилл в БД либо явное исключение для админов.

**Фиксы:**
- **Бэкофилл (адресует «даже админ»):** `UPDATE users SET email_verified = true WHERE role = 'admin';` в gateguard БД (и/или для seed-аккаунтов).
- **Опционально — обход для админа** во `verified_gate.go` и баннере (продуктовое решение).
