## 1. Как читать документ

Каждый сценарий проверяем по **двум осям**:

- **Ф (функционал)** — работает ли технически, правильные ли статусы/данные/ответы.
- **Д (дизайн/UX)** — понятно ли пользователю, приятно ли, нет ли тупика, читается ли на мобиле и в тёмной теме, адекватны ли пустые состояния и тексты ошибок (весь UI — на русском).

Формат: **Цель → Шаги → Ожидаемо (Ф) → Смотрим глазами (Д) → Риски/что уже сломано.**

Легенда статусов результата для прогона: ✅ ок · ⚠️ работает, но есть замечание · ❌ баг/блокер · 🚧 фичи нет.

## 2. Подготовка окружения (сделать до прогона)

1. **Тестовые аккаунты** (регистрация email+пароль, ≥8 символов):
   - `org-tarsky@…`, `org-friend1@…`, `org-friend2@…`, `org-friend3@…` — организаторы.
   - `guest1@…` … `guest4@…` — «аудитория» (записываются, оставляют заявки).
   - Реальный админ — `poulissimo@gmail.com` (роль в БД gateguard). Промоушен ролей — только руками через SQL, UI нет.
2. **Проверить, что демо-логин выключен в prod** (`HTTP_MOCK_AUTH=false`) — иначе любой email логинится без пароля. Это контроль compliance, на живых клиентах должен быть off.
3. **Данные для B2 (Геленджик)** теперь заводятся **прямо из формы** (R1 задеплоен): режим «По заявке» + лимит 10 + кураторский вопрос — см. S8.1. Отдельное API-сидирование больше не нужно (кроме S8.3, где нужно завершившееся событие с прошедшей датой).
4. **Устройства**: минимум iPhone Safari + Android Chrome + десктоп Chrome; проверить **тёмную и светлую** тему (переключатель солнце/луна). Таймзона отображения зафиксирована `Europe/Moscow`.
5. **Мониторинг на время пилота**: Prometheus доступен по SSH-туннелю `ssh -L 9091:localhost:9091 vdska2` → `127.0.0.1:9091`. Держать открытым дашборд RED (ошибки/латентность) во время первых заходов орг.

## 3. ✅ Бывшие критические риски R1–R3 — РЕАЛИЗОВАНО и задеплоено 2026-07-14

> Первоначально этот блок был списком блокеров. Все три **закрыты фичами** (ветка `feat/organizer-suite-r1-r2-r3`, миграция БД `019`, задеплоено на **`presence.tarski.ru`** 2026-07-14, прод-проверено: детальная страница события отдаёт 200 с реальными данными). Ниже — не «риски», а **как проверить каждую фичу на живом сайте**. Подробные пошаговые сценарии — в новом **Блоке 8**. R4 ниже остаётся известным ⚠️.

### R1 ✅ Конфигурация записи прямо в форме создания
Форма `/events/new` теперь имеет секцию **«Запись»**: сегмент-контрол **Открытая / По заявке / Внешняя ссылка**, поле **Лимит мест**, **Вопрос кандидату** (для заявок), **Ссылка для регистрации** (для внешней). Тур «до 10 по отбору» (B2 Геленджик) заводится целиком из UI.
- **Как проверить на сайте:** `/events/new` → секция «Запись» → «По заявке» → Лимит 10 + вопрос «Над чем работаете?» → опубликовать → на детальной видно «осталось N мест», гость подаёт заявку, организатор принимает/отклоняет в `/organizer/applications`. Прогнать все три режима + пустой лимит (без ограничения). См. **S8.1**.
- **Валидация:** заявка без вопроса / внешняя без URL / лимит ≤ 0 → русское сообщение, сабмит блокируется.

### R2 ✅ Редактирование опубликованного события
Появилась страница `/events/[id]/edit` и кнопка **«Редактировать»** на `/events/mine` (только для `draft`/`published`). Правка контента/даты/места/цены/обложки/категорий/**вместимости**; событие остаётся published; правки пишутся в `audit_log`.
- **Как проверить на сайте:** опубликовать событие → «Редактировать» → изменить название и перенести дату → сохранить → остаётся опубликованным, изменения видны. Поднять лимит при наличии листа ожидания → участник из waitlist получает место. Опустить лимит ниже занятого → русское сообщение «Нельзя уменьшить лимит мест ниже числа уже записавшихся» (не английский текст). Режим записи после публикации — заблокирован с подсказкой. См. **S8.2**.

### R3 ✅ Приватные отзывы участников после события
Новый домен отзывов: участник с активным RSVP на **завершённом** событии ставит **★ (1–5) + необязательный комментарий** (1 отзыв/чел); организатор видит их приватно (средняя + список, **имя без email**).
- **Как проверить на сайте:** нужно **завершившееся** событие с участником (сидировать `starts_at`/`ends_at` в прошлом + RSVP `going`). Участник на детальной странице или в `/me/practices` (прошедшие) ставит оценку → «Спасибо за отзыв 🙌». Организатор видит средний балл и отзывы в `/events/mine`; чужой/аноним — не видит (403 → пусто). См. **S8.3**.
- **Обходной путь для расширенных опросов** («что улучшить» свободным текстом сверх ★) остаётся внешняя форма в описании события — на старте достаточно ★+комментарий.

### R4. `my_rsvp_status` не отдаётся на детальной странице
`GET /events/{id}` не заполняет статус текущего пользователя. Вернувшийся участник на свежей загрузке видит кнопку «Записаться», как будто ещё не записан; корректное состояние появляется только после действия/в рамках сессии.
- **Кого бьёт:** любого, кто вернулся посмотреть «я точно записан?» — типичное поведение перед событием.
- **Проверка:** записаться → обновить страницу (или зайти с другого устройства) → посмотреть состояние кнопки. Ожидаемо (сейчас): кнопка не помнит запись. Отметить как ⚠️.

## 4. Сценарии

### Блок 1 — Онбординг организатора

#### S1.1 Регистрация и вход
- **Цель:** орг заводит аккаунт.
- **Шаги:** «Войти» → регистрация email+пароль (≥8) → выйти → войти повторно → неверный пароль.
- **Ф:** register → 200; повтор того же email → 409; login верный → 200; неверный → 401.
- **Д:** понятно ли, где регистрация vs вход (тумблер в модалке)? Сообщение об ошибке пароля/занятого email — по-русски, человеческое? Модалка не «прыгает» при гидрации? На мобиле клавиатура не перекрывает кнопку?
- **Риски:** email-верификация — заглушка (письмо только логируется). Значит «подтвердите почту» пользователь не получит — не обещать этого оргам.

#### S1.2 Профиль организатора + верификация
- **Цель:** орг оформляет бренд и проходит верификацию (для галочки ✓ и публичной страницы).
- **Шаги:** `/me/organizer` → заполнить профиль → «Отправить на верификацию» → статус `pending`. Затем админ (`poulissimo`) в `/admin/moderation/organizers` → verify. Проверить resubmit после reject и revoke.
- **Ф:** draft→pending→verified; публичная `GET /organizers/{id}` для непроверенного → 404 (нет утечки); reject/revoke без причины → 400. Если включён `organizers.auto_verify_all` — submit сразу verified.
- **Д:** видно ли оргу текущий статус и причину отклонения? Понятно ли, что даёт верификация? Пустой профиль до заполнения — адекватное состояние?
- **Риски:** админ-путь верификации **ещё не проверялся на prod** живым админом (по HANDOFF). Прогнать реально до пилота, иначе орги зависнут в `pending`. Решить заранее: включаем `auto_verify_all` на время пилота или админ верифицирует руками.

#### S1.3 Публичная страница организатора
- **Цель:** гость видит организатора и его события.
- **Шаги:** `/organizers/[id]` для verified орга с ≥1 опубликованным событием.
- **Ф:** список **предстоящих** (основной) + **прошедших** (до ~10) опубликованных событий; кнопка подписки; для неверифицированного/несуществующего — не палим данные.
- **Д:** ⚠️ страница `use client` — контент рисуется после загрузки в браузере, SSR-HTML пуст. Проверять **только в реальном браузере**, не curl. Смотрим: не мигает ли пустотой, есть ли скелетон/лоадер, как выглядит орг без прошедших событий.

### Блок 2 — Создание событий (по реальным кейсам Tarsky)

#### S2.1 Рыбалка B1 — happy path через форму (open, бесплатно, гео)
- **Цель:** базовый сценарий, который пройдут почти все орги.
- **Шаги:** `/events/new` → обложка (≤5 МБ, картинка) → заголовок «Рыбалка с художниками» → описание → категории (мультивыбор) → формат «оффлайн» → **место**: через `VenuePicker` найти точку на Ладоге/канале (Nominatim-поиск адреса + пин на карте) или создать новое → **начало** (обязательно) → окончание → бесплатно → «Опубликовать».
- **Ф:** POST → 201, статус published; событие появляется в ленте `/` и на `/map`; гео-точка корректна (в `/map` и «рядом»).
- **Д:** форма понятна без инструкции? Загрузка обложки даёт превью и прогресс? Поиск адреса находит нецентровые точки (Ладога — это не адрес, а зона; проверить, что пин можно поставить руками)? Сообщения валидации по-русски? На мобиле карта-пикер юзабельна пальцем?
- **Риски:** место на воде/в природе может не иметь адреса — критично, чтобы работал ручной пин, а не только текстовый поиск.

#### S2.2 Тур в Геленджик B2 — многодневное, лимит 10, отбор ✅ (было БЛОКЕРОМ, решено R1 → см. **S8.1**)
- **Цель:** самый сложный и показательный кейс. _Историческая запись: до 2026-07-14 формат не создавался формой. Теперь режим «По заявке» + лимит + вопрос есть в форме (R1) — актуальные шаги в S8.1. Ниже — исходный обходной путь через API, если нужен._
- **Шаги (через форму):** попытаться завести событие с датами 2–3 дня, лимитом 10 и отбором.
- **Ф ожидаемо (сейчас):** ❌ форма **не даёт** задать лимит мест и режим заявок — создастся open/без лимита. Многодневность — `start`+`end` форма поддерживает, проверить, что `end` на др. день сохраняется и корректно показывается.
- **Обход для пилота (API):** `POST /api/v1/events` с телом, включающим:
  ```json
  { "title":"Тур в Геленджик", "starts_at":"…", "ends_at":"…(+2–3 дня)",
    "signup_mode":"application", "capacity":10,
    "curator_question":"Расскажите, над чем сейчас работаете",
    "status":"published", "format":"offline", "price_type":"free" }
  ```
  → затем проверить сценарий отбора в S4.2 и управление в S5.1.
- **Д:** как многодневное событие выглядит в ленте, на детальной, в календаре (одна дата или диапазон)? Понятно ли участнику, что это 2–3 дня? Для «до 10» — видит ли гость, что мест ограничено и сколько осталось (`seats_remaining`)?
- **Решение:** зафиксировать — либо чиним форму (R1), либо этот тур заводит не сам организатор, а мы через API и отдаём ему ссылку на `/organizer/applications`.

#### S2.3 Пинг-понг СПб B3 — казуальное open
- **Цель:** лёгкое городское событие, СПб-гео.
- **Шаги:** как S2.1, место — конкретный адрес в СПб, категория «игры/спорт» (что есть в списке).
- **Ф/Д:** проверить, что таксономия категорий покрывает такой формат; если подходящей категории нет — это сигнал (таксономия-админ #5 ещё не построена, менять категории на лету нельзя).

#### S2.4 События дружественных орг (волна A)
- **Цель:** каждая из 3 орг заводит 2–3 события подряд.
- **Ф:** проверить **квоту 10 событий/мес** на организатора (граница по `Europe/Moscow`): 11-е → 429 с русским текстом «лимит 10 событий в месяц». Для пилота 2–3 события в лимит укладываются — но проверить, что текст лимита адекватен, если кто-то нагенерит.
- **Д:** удобно ли заводить несколько событий подряд (форма сбрасывается? есть ли «создать ещё»?). Список `/events/mine` и `/organizer` читаемо показывает счётчики draft/published?

#### S2.5 Черновик → превью → публикация
- **Цель:** орг готовит событие заранее.
- **Шаги:** создать со статусом **Черновик** → зайти на `/events/[id]` своей ссылкой (владелец видит черновик через `OwnerDraftFallback`) → из `/events/mine` нажать «Опубликовать».
- **Ф:** черновик не виден в публичной ленте и гостям (гость на ссылку черновика → 404/не видит); владелец — видит превью; publish (`PATCH status:published`) → появляется в ленте.
- **Д:** понятно ли, что событие ещё черновик (бейдж)? Есть ли явная кнопка «Опубликовать» и подтверждение? Не публикуется ли случайно (дефолт формы — «Опубликовать», проверить, что осознанно).

#### S2.6 Медиа и контент-краевые случаи
- Обложка >5 МБ → отказ с понятным текстом; не-картинка → отказ (MIME-allowlist).
- Очень длинное описание/эмодзи/переносы — не ломают вёрстку карточки и детальной.
- Событие без обложки — есть ли аккуратный плейсхолдер?

### Блок 3 — Аудитория находит событие

#### S3.1 Лента (Discovery)
- **Ф:** на `/` только `published`; черновики/снятые не видны.
- **Д:** первые впечатления гостя — понятно, что это, когда, где, сколько? Карточка читается на мобиле? Пустая лента (если в городе гостя ничего нет) — адекватное состояние?

#### S3.2 Поиск
- **Ф/Д:** найти событие Tarsky по названию/ключу. ⚠️ **AI-поиск — заглушка**; проверить, что базовый поиск работает и не обещает «умного» поведения, которого нет.

#### S3.3 Карта и «рядом» — мультигород (СПб / Геленджик / Ладога)
- **Цель:** события в разных городах — ключевой стресс для гео.
- **Шаги:** `/map`, «рядом со мной» из СПб; проверить, что событие в Геленджике **не** попадает в «рядом» из СПб (радиус ≤50 км), но находится на карте при перемещении.
- **Ф:** `distance_m` считается; PostGIS-радиус 50 км; событие на Ладоге показывается за пределами центра.
- **Д:** маркеры не крашат карту (был баг с иконкой Leaflet — проверить, что пофикшено); кластеризация/читаемость при событиях в разных городах; понятно ли, что показывается по гео, а что — вся страна.

#### S3.4 Фильтр по категориям
- **Ф/Д:** фильтр по категории события Tarsky возвращает его; нет дубля чипа «Рядом» (был баг). Если у B1/B3 нет точной категории — фиксируем пробел таксономии.

### Блок 4 — Участие аудитории

#### S4.1 Open-запись + лист ожидания (B1, B3)
- **Шаги:** гость на событии → «Записаться» → (если сидировано с capacity) заполнить места → следующий → «В лист ожидания». Отписаться → проверить FIFO-промоушен из waitlist в going.
- **Ф:** свободно → `going`; полно → `waitlist`; отмена going → самый старый waitlist промоутится в going в одной транзакции; счётчик `seats_remaining` корректен под нагрузкой (лок `FOR UPDATE`).
- **Д:** тексты кнопок «Записаться»/«В лист ожидания»/«Отписаться»/«Покинуть лист» понятны? Видно ли, сколько мест осталось? После записи — есть подтверждение?
- **Риск R4:** обновил страницу — кнопка забыла, что я записан (⚠️).

#### S4.2 Заявки/отбор (B2 Геленджик) + решения организатора
- **Шаги (нужно сидированное application-событие из S2.2):** гость → «Подать заявку» → ответ на кураторский вопрос → `applied`. Организатор в `/organizer/applications` или на `/events/mine` (инлайн-панель) видит заявку с **именем** (не email) и **счётчиком** → «Принять»/«Отклонить».
- **Ф:** applied → accept: если есть места → `accepted`, если нет → авто-`waitlist`; decline → `declined`; повторное решение по уже решённой заявке → 409; заявка на чужое событие → 404. Гость может «Отозвать заявку» (cancel).
- **Д:** организатору понятно, кто подал и что ответил? Виден ли лимит 10 и сколько принято? Участнику понятен статус его заявки (`/me/applications`)? Кураторский вопрос показывается корректно?
- **Приватность:** проверить, что организатору отдаётся **имя, но не email** заявителя (это специально так).

#### S4.3 Внешняя регистрация (external)

- **Ф:** бэкенд отдаёт 422 с URL, фронт открывает ссылку. **Д:** понятно ли, что регистрация уходит на внешний ресурс (а не баг)?

#### S4.4 Экспорт .ics (в календарь телефона)

#### S4.5 Личный календарь `/me/calendar`
- **Ф:** месяц/неделя/день; два потока с цветовой легендой — «я иду» (активный RSVP) + «от подписок» (события орг, на которые подписан); событие может быть в обоих. Границы дня — по `Europe/Moscow`.
- **Д:** легенда понятна? Многодневный тур B2 показывается на всех днях диапазона? Пустой календарь — адекватен?
- **Риск:** визуальный цветовой поток «иду/подписка/оба» **не проверялся на prod** — нужен verified орг + залогиненный юзер.

#### S4.6 Подписка на организатора + лента
- **Ф:** `/organizers/[id]` → подписаться/отписаться; события подписанного орга влетают в календарь (`from_followed`). Подписка на неверифицированного орга → 404.
- **Д:** ⚠️ карточка «Подписчики» в хабе организатора **отключена** («позже») — организатор не видит своих подписчиков. Не обещать оргам эту цифру.

#### S4.7 «Мои записи» и «Мои заявки»

### Блок 8 — Проверка релиза R1/R2/R3 на живом сайте (задеплоено 2026-07-14)

> Прогонять на **https://presence.tarski.ru**. Нужны: 1 аккаунт-организатор + 2 гостя. Для S8.3 нужно **завершившееся** событие с участником — проще всего засидировать через API/БД (событие с `ends_at` в прошлом + RSVP `going`), т.к. ждать реального завершения долго.

#### S8.1 R1 — режимы записи и лимит в форме создания
- **Шаги:** войти организатором → `/events/new` → долистать до секции **«Запись»**.
  1. **По заявке + лимит 10:** выбрать «По заявке», Лимит = 10, Вопрос кандидату = «Над чем сейчас работаете?» → опубликовать. Это ровно кейс B2 Геленджик.
  2. **Открытая + лимит:** новое событие, «Открытая», Лимит = 3.
  3. **Внешняя ссылка:** «Внешняя ссылка» → поле URL появляется, лимит/вопрос скрыты → указать `https://…`.
- **Ф:** событие «по заявке» на детальной показывает «осталось 10 мест»; гость видит «Подать заявку»; ответ на вопрос сохраняется; организатор видит заявку с именем в `/organizer/applications`. Открытая с лимитом 3: 4-й записавшийся → лист ожидания. Внешняя: CTA открывает URL в новой вкладке.
- **Валидация (Д):** «По заявке» без вопроса / «Внешняя» без URL / Лимит = 0 → русское сообщение под полем, сабмит блокируется.
- **Регрессия:** обычное «Открытая, без лимита» (пустое поле) создаётся как раньше.

#### S8.2 R2 — редактирование опубликованного события
- **Шаги:** организатор → `/events/mine` → у **опубликованного** события кнопка **«Редактировать»** → `/events/[id]/edit`.
  1. Изменить название + перенести дату/время → сохранить → **остаётся опубликованным**, изменения видны на детальной и в ленте.
  2. **Лимит вверх при листе ожидания:** событие с лимитом 3 и заполненным waitlist → поднять лимит до 5 → участники из листа ожидания получают места (going).
  3. **Лимит вниз ниже занятого:** попытаться опустить лимит ниже числа записавшихся → русское сообщение **«Нельзя уменьшить лимит мест ниже числа уже записавшихся»** (не английский текст, не про capacity).
  4. **Блокировка режима:** сегмент «Запись» на published задизейблен с подсказкой «Режим записи зафиксирован после публикации».
- **Ф:** событие после правки — по-прежнему `published`; правки пишутся в `audit_log`. Кнопки «Редактировать» **нет** у `pending_review`/`rejected`/`cancelled` строк.
- **Д:** при смене даты/места — предупреждение «Участники уже записаны — предупредите их об изменении самостоятельно».

#### S8.3 R3 — приватные отзывы после события
- **Подготовка:** засидировать (API/БД) **завершившееся** событие (`ends_at` в прошлом) + у гостя RSVP `going`/`accepted` на нём.
- **Шаги (участник):** войти гостем → открыть это событие (или `/me/practices` → вкладка «Прошедшие») → блок **«Как всё прошло?»**: выбрать ★ (1–5) + при желании комментарий → отправить → **«Спасибо за отзыв 🙌»**. Обновить страницу → показывается состояние «уже оставлен» (форма не дублируется).
- **Шаги (организатор):** войти владельцем события → `/events/mine` → раскрыть событие → блок **«Отзывы»**: средняя ★ + количество + список (оценка, комментарий, **имя автора без email**, дата).
- **Ф / приватность:** чужой организатор или аноним по `GET /events/{id}/feedback` → 403/401, в UI ничего не видно (не палится, что отзывы есть). Повторный отзыв тем же участником → 409 «Вы уже оставили отзыв». Отзыв до завершения события → 422. Не-участник → 403 «Отзыв могут оставить только участники».
- **Д:** ★-виджет крупный, тач-дружелюбный; неавторизованному на завершённом событии — приглашение войти (не пустой блок); у организатора без отзывов — «Отзывы появятся после завершения события».

#### S8.4 Регрессия по критичному пути (обязательно)
- Открыть **детальную страницу любого опубликованного события** в браузере (`/events/{id}`) — должна грузиться **200 с реальными данными** (не «Server error»). _Ранее релиз ловил 500 на этом пути — этот шаг обязателен после каждого деплоя фронтенда._
- Для **будущего** (не завершившегося) события кнопка записи/лист ожидания работает как раньше (блок отзывов не появляется).

## 5. Сквозной продуктово-дизайнерский чек-лист (применять к каждому экрану)

- **Мобайл-фёрст:** iPhone Safari + Android Chrome; таб-бар не перекрывает контент (`max-sm:pb-28`), кнопки достаются большим пальцем, карта/пикер работают тач-жестами.
- **Тёмная/светлая тема:** переключатель солнце/луна; контраст текста, читаемость бейджей и карт в обеих темах; нет «вспышки» неправильной темы при загрузке.
- **Пустые состояния:** лента без событий, организатор без событий, пустой календарь, ноль заявок — везде осмысленный текст, а не пустой экран.
- **Ошибки и копирайтинг:** все ошибки по-русски и человеческим языком (401/403/404/409/429/503 не должны утекать как сырой код/стек). Лимит 10/мес, «недостаточно прав», «событие нельзя редактировать» — понятные формулировки.
- **Навигация назад:** на каждой вторичной странице есть «‹ События»/возврат; из глубоких экранов нет тупиков.
- **Состояния загрузки:** `use client`-страницы (`/organizers/[id]`, календарь) не мигают пустотой — есть скелетон/лоадер.
- **Время и таймзона:** все даты в `Europe/Moscow`, нет hydration-mismatch по датам; многодневные события отображают диапазон.
- **Приватность на публичных поверхностях:** нигде не светится email (в организаторе события, в заявках — только имя).
- **Производительность:** во время первых заходов орг держать дашборд Prometheus RED; следить за 5xx и латентностью create/list/detail.
